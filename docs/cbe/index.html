Compute Batch Environent (CBE)

CBE Access and Quick User Guide

Important notice!

Scheduled preventive maintenance window on the CBE compute cluster:

  • EVERY first Thursday of every month at 10 a.m. until 10 a.m. on Friday

This document is a short quick acces guide covering the most basic SLURM commands for CBE.

Access

You can reach the clusters via ssh using your institute account.

The Login nodes are reachable via SSH from our network, if you want to connect from another location you have to connect to our Network via VPN first.

ssh cbe.vbc.ac.at

Login nodes

CBE has two login nodes. ssh to cbe.vbc.ac.at address will take you in a round-robin fashion to one of them. The individual login nodes can be SSH’d into via hostnames clip-login-{0,1}.cbe.vbc.ac.at.

Jobs on login nodes

Never run your jobs directly on a login node
The SLURM job submission system will distribute the workload among the machines, for more information about job submission, refer to the quick-start documentation below and our SLURM Advanced User’s Guide.

Brief overview of the cluster

Cluster in general consists of two types of nodes (with some subtypes included, see: CLIP.

  • login nodes - 2x
    • To submit jobs, compile software for the cluster, check your job status, etc. you connect to the ‘Login Nodes’ - they are your gateway to the cluster. These are nodes with additional software (e.g. development libraries) and these nodes are not used for computation.
  • compute nodes - up to 216x
    • Jobs submitted from login nodes get distributed (via. SLURM job scheduler) to compute nodes where they execute. Direct ssh access to compute nodes has beed disabled to prevent (un)intentional interference with running jobs

To provide environment for efficient work, CBE cluster also provides:

  • several special storage areas, see CBE Storage User’s Guide
  • bundle of preinstalled software, see CBE Application and Software User’s Guide

Running a job on the cluster

CBE utilizes SLURM: “open source, fault-tolerant, and highly scalable cluster management and job scheduling system”

As a cluster resource manager (resources being: CPUs, memory, custom resources like SSDs, etc.), SLURM has three key functions:

  1. Allocates exclusive and/or non-exclusive access to resources (compute nodes) to users for some duration of time so they can perform work.
  2. Provides a framework for starting, executing, and monitoring work (normally a job) on the set of allocated node or for parallel jobs on nodes.
  3. Finally, it arbitrates conflicting requests for resources by managing a queue of pending work.

SLURM Terminology

The entities managed by these SLURM include nodes, the compute resource in SLURM, partitions, which group nodes into logical disjoint sets, jobs, or allocations of resources assigned to a user for a specified amount of time, and job steps, which are sets of (possibly parallel) tasks within a job. Each job in the priority-ordered queue is allocated nodes within a single partition.

The notion of array jobs which is a feature useful for issuing big numbers of jobs with common characteristics is supported through the concept of job steps.

Once a job is assigned one node or a set of nodes, the user is able to initiate work in the form of job steps in any configuration within the allocation. For instance, a single job step may be started that utilizes all nodes allocated to the job, or several job steps may independently use a portion of the allocation.

SLURM Command Line Utilities

Users interact with SLURM through four command line utilities

  • salloc is used to allocate resources for a job in real time. Typically this is used to allocate resources and spawn a shell. The shell is then used to execute srun commands to launch parallel tasks.
  • sattach is used to attach standard input, output, and error plus signal capabilities to a currently running job or job step. One can attach to and detach from jobs multiple times.
  • sbatch is used to submit a job script for later execution. The script will typically contain one or more srun commands to launch parallel tasks.
  • sbcast is used to transfer a file from local disk to local disk on the nodes allocated to a job. This can be used to effectively use diskless compute nodes or provide improved performance relative to a shared file system.
  • scancel is used to cancel a pending or running job or job step. It can also be used to send an arbitrary signal to all processes associated with a running job or job step.
  • scontrol is the administrative tool used to view and/or modify SLURM state. Note that many scontrol commands can only be executed as user root.
  • sinfo reports the state of partitions and nodes managed by SLURM. It has a wide variety of filtering, sorting, and formatting options.
  • smap reports state information for jobs, partitions, and nodes managed by SLURM, but graphically displays the information to reflect network topology.
  • squeue reports the state of jobs or job steps. It has a wide variety of filtering, sorting, and formatting options. By default, it reports the running jobs in priority order and then the pending jobs in priority order.
  • srun is used to submit a job for execution or initiate job steps in real time. srun has a wide variety of options to specify resource requirements, including: minimum and maximum node count, processor count, specific nodes to use or not use, and specific node characteristics (so much memory, disk space, certain required features, etc.). A job can contain multiple job steps executing sequentially or in parallel on independent or shared nodes within the job’s node allocation.

Working with SLURM – Basics

Every job in SLURM is described with an sbatch script. In essence, it is a regular shell script augmented with #SBATCH stanzas, as you will see below.

This is an example submission script that shows basic SLURM concepts:

#!/usr/bin/env bash

# Lines beginning with #SBATCH are instructions for SLURM, this line is a comment :)
# For example:

# This job requests from SLURM to allocate 1 node.
#SBATCH --nodes=1
# On that node, it will run 4 tasks, each with 1 core and 1 GB of memory.
#SBATCH --ntasks=4
#SBATCH --cpus-per-task=1
#SBATCH --mem-per-cpu=1G
# Running time will be 2 minutes#SBATCH --time=0-00:02:00
# And it will place the output of the commands into my.stdout file
#SBATCH --output=my.stdout

# Total allocation will be: 4 cores and 4GB of memory on 1 node.

# Executing a command like this will cause it to run once in the whole allocation:
/bin/hostname

# These commands will be run by each task (4 times total)
# -l flag prepends the task number to each line of output
srun -l /bin/hostname
srun -l /bin/pwd

# an we will sleep for 60 seconds before exiting
sleep 60
# after job is finished, see file my.stdout for results

Submission

Scripts are submitted to SLURM with sbatch command:

sbatch simple.sbatch
Submitted batch job 145843

After submission, SLURM will evaluate the script, check if there are enough available resources (cpu, memory, nodes, etc.) on the cluster to satisfy the requests from the script and if there are, place the job on the nodes for execution.

Number in red is the SLURM_JOB_ID, incrementally increasing unique number slurm assigns to every job.

(see later: cancelling the job)

Interactive sessions can be started the following way

[john.doe@clip-login-1 ~]$ srun --pty bash
[john.doe@clip-c2-6 ~]$ hostname
clip-c2-6

Monitoring Jobs

All the jobs running on the cluster, also the ones that are waiting in the “queue” for free resources can be observed with the squeue command:

without any flags, it will display all the jobs on the cluster:

[john.doe@clip-login-1 ~]$ squeue 
JOBID PARTITION     NAME     USER ST       TIME  NODES NODELIST(REASON)
10          c     bash jane.doe  R       0:51      1 clip-c2-0
 7          c     bash john.doe  R      30:20      1 clip-c2-0

Or, if you are interested only in your jobs, add -u your_username switch:

[john.doe@clip-login-1 ~]$ squeue -u john.doe
JOBID PARTITION     NAME     USER ST       TIME  NODES NODELIST(REASON)
7         c     bash john.doe  R      31:08      1 clip-c2-0

Cancelling the job

If for some reason you wish to cancel submitted job, scancel is the command.

It can cancel all the jobs from the user with -u your_username or you can target a specific job with the SLURM_JOB_ID number.

e.g. you wish to cancel ALL your submitted jobs

[john.doe@clip-c2-0 ~]$ squeue -u john.doe
            JOBID PARTITION     NAME     USER ST       TIME  NODES NODELIST(REASON)
                13         c     bash john.doe  R       0:47      1 clip-c2-0
                12         c     bash john.doe  R       0:50      1 clip-c2-0
                11         c     bash john.doe  R       3:10      1 clip-c2-0
[john.doe@clip-c2-0 ~]$ scancel -v -u john.doe

scancel: Cray node selection plugin loaded
